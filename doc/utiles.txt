PASO 2: Procesamiento de documentos — doc_chunking.py
La función principal es get_text_split(). Recibe el nombre del blob y orquesta todo el pipeline para un documento. Veamos sub-paso a sub-paso:

2.1 Descarga del Blob Storage (líneas 62-66)

file_download = asyncio.run(get_specific_file_from_blob_container(
    storageaccountname, storageaccountkey, containername, docId, ...
))

Usa azure_storage_service.get_specific_file_from_blob_container() que:

Conecta con AsyncBlobServiceClient
Descarga el blob como bytes (blob_bytes)
Tiene 3 reintentos con delays de 2, 5 y 10 segundos
2.2 Extracción con Document Intelligence (líneas 68-72)
docintelligence_service.get_content_from_document():

result = asyncio.run(get_content_from_document(
    file_download=file_download, boolformulas=boolformulas
))

Conecta con DocumentIntelligenceClient de Azure
Usa el modelo prebuilt-layout (extrae tablas, párrafos, fórmulas, idiomas)
Formato de salida: markdown (output_content_format="markdown")
Si es PDF con fórmulas: activa features FORMULAS + LANGUAGES
Tiene 3 reintentos automáticos
Devuelve un result con: .content (markdown completo), .paragraphs, .pages, .tables, .languages
2.3 Etiquetar páginas (líneas 77-152)
Recorre result.paragraphs (para PDFs) o result.pages (para otros formatos) y genera un diccionario page_contents[nº_página] con los párrafos de cada página.

Después, inserta marcadores de página (PG1, PG2, ...) dentro del full_content al final de cada página. Estos marcadores se usan luego para saber a qué páginas pertenece cada chunk.

2.4 Procesamiento de fórmulas (líneas 155-196)
Si boolformulas=True:

Extrae fórmulas por página de result.pages[i].formulas
Reemplaza marcadores :formula: en el contenido con las fórmulas matemáticas reales
Limpia etiquetas de fórmula residuales
2.5 Procesamiento de tablas (líneas 199-285)
Complejo proceso:

Convierte cada tabla a markdown con processchunks.convert_table_to_markdown(table)
Detecta tablas que son footers (aparecen como última tabla en >50% de páginas) y las descarta
Combina tablas que continúan entre páginas (mismo nº columnas, páginas consecutivas)
Genera tables_by_page[nº] — lista de tablas markdown por página
2.6 Limpieza de ruido (líneas 291-321) — Timestamp "06 removeContentNoise"
Aplica 14 filtros de limpieza en cadena, usando funciones de processchunks:

Función         	                    Qué hace
clean_false_checkbox()	                Elimina checkboxes con confianza < 70%
detect_and_remove_headers_footers()	    Elimina líneas que se repiten ≥7 veces (headers/footers)
eliminar_figuras_repetidas()	        Elimina <figure> duplicadas
detect_and_remove_duplicate_tables()	Elimina tablas duplicadas (>20% de páginas)
+ 10 regex	                            Limpia: checkboxes masivos, fórmulas residuales, <figure> vacías, comentarios HTML, PageHeader, PageFooter, FigureContent, marcadores MIDDLE, etc.
2.7 Chunking (línea 323) — Timestamp "07 ChunkingProcess"

chunks = processchunks.markdown_percentage(filtered_full_content, 2000, 0.1, 1000, 3000)

Parámetros: chunk_size=2000, overlap=10%, min_tokens=1000, max_tokens=3000.

Detalle de markdown_percentage() en processchunks.py:424-468:

Divide por encabezados markdown (H1-H5) usando MarkdownHeaderTextSplitter de LangChain
Combina fragmentos pequeños (<min_tokens=1000 tokens) con un buffer acumulador
Combina tablas cercanas con combinar_splits_tablas() — no separa tablas que estén a <50 chars de distancia
Subdivide fragmentos grandes (>4000 tokens) con dividir_si_excede_max_tokens():
Si tiene tabla → dividir_chunk_con_tablas() (no rompe tablas)
Si no tiene tabla → dividir_con_solapamiento() (10% overlap)
2.8 Enriquecimiento de cada chunk (líneas 340-455)
Para cada chunk se:

Extrae páginas con regex PG(\d+) → genera page_range (ej: "3-5")
Detecta headers (H1-H5) con detect_title(), detect_subtitle(), etc.
Hereda headers del chunk anterior: si un chunk no tiene header propio, se le prepende el último header conocido (para dar contexto)
Asocia tablas y fórmulas por rango de páginas
Llama a processchunks.upload_chunks() pasando todo
2.9 Upload de cada chunk a Cosmos DB — upload_chunks() en processchunks.py:80-148
Para cada chunk:

Soft-delete previo (línea 91): Si es el primer chunk del doc (index==1), llama a mark_existing_chunks_as_deleted() que busca en Cosmos todos los chunks anteriores del mismo docTitle y les pone isDeleted=True

Genera ID: UUID sanitizado para Azure Search

Limpia marcadores de página del contenido (PG\d+)

Genera chunkId: hash SHA-256 de docTitle + content + title + page

Genera embedding (línea 140):
get_embedding(embeddingcontent, Config.azure_openai_emb_name, ...)

Llama a openai_service.get_embedding() que usa AzureOpenAI.embeddings.create() con el modelo ada-002 → vector de 1536 dimensiones

Construye el documento JSON con todos los campos que coinciden con el schema del índice de Azure Search

Sube a Cosmos DB con upload_doc_cosmos() → hace container.upsert_item(data)

PASO 3: Azure Search Indexer (sincronización)

Construir/actualizar el índice de búsqueda híbrida

¿QUÉ FALTA?
1. Fichero orquestador (runner) — NO EXISTE
No hay ningún script que diga "recorre todos los blobs del container y llama a get_text_split() para cada uno". Necesitas un fichero tipo run_ingestion.py que:

Liste los blobs del container (la función list_json_configs_from_blob ya existe pero para JSONs; necesitas una versión genérica)
Para cada blob, llame a get_text_split(blob_name, docId, ...)
Gestione paralelismo (max_workers_docs=3 ya está en config)
Los ficheros run_cvs.py, run_eu.py, run_wikipedia.py existen pero dijiste que los ignoremos.

2. Config.cosmosdb_container — NO EXISTE en el config
En doc_chunking.py:21 se usa Config.cosmosdb_container pero en config.py solo existe cosmosdb_container_cvs. Esto fallará en runtime. O se necesita añadir cosmosdb_container al config, o usar cosmosdb_container_cvs.

3. Campos QuestionsText y docSummary siempre vacíos
El índice de Azure Search los define como searchable y semánticos, pero upload_chunks() siempre los envía como "". Si quieres búsqueda semántica sobre preguntas generadas o resúmenes, necesitarías un paso adicional con GPT que genere esos campos.

4. El indexer create_indexer() está comentado
En create_index.py:200 la llamada está comentada. Deberás descomentarla o crear el indexer manualmente desde el portal de Azure.

5. datapoints_list.json probablemente no existe
load_datapoints() busca un fichero datapoints_list.json en el directorio de trabajo. Si no existe, devuelve None y los datapoints se dejan vacíos (no es un error, simplemente no se extraen keywords).

¿Quieres que cree el fichero orquestador run_ingestion.py que falta, o prefieres profundizar en algún paso concreto?
